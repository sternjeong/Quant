## 1. 요청 요약 (2026-07-14, 대화로 확정됨)

임의의 유튜브 주식 전략 채널이 소개하는 전략을 텍스트로 입력받아:

1. S&P500 중 약 100개 종목에 한 번에 백테스팅한다 (기존 자연어 전략 해석기 재사용, 대상만 100종목으로 확장).
2. 유튜버의 원본 전략을 **백본으로 유지**하면서(강화학습 X, 구조 재설계 X) 파라미터를 자동으로
   미세튜닝(grid/random search)하는 엔진을 만든다.
3. 각 종목이 어떤 **투자 스타일/유형**(성장주, 주도주 등 — 사용자가 아는 범위를 넘어 합리적인
   전체 체계를 설계해달라고 요청함)에 속하는지 스스로 판별하고, 유형에 맞게 **파라미터 탐색
   방향을 다르게** 튜닝한다.
4. 100종목 중 **상위 N개**(N은 사용자가 정함)를 뽑아 보여주되, 기본 정렬 기준은 **초과수익
   (전략 CAGR − S&P500 매수보유 CAGR)**, 사용자가 직접 종목을 선택/변경할 수도 있어야 한다.
5. 결과 화면은 종목별로 **전략 적용 vs 해당 종목 매수보유 vs S&P500 매수보유** 3-way 비교로 보여준다.
6. 바로 코드 작성 X — 충분히 상의 후 진행, 대화는 이 md 파일로 정리하며 진행.

## 2. 재사용 가능한 기존 자산 (코드 확인 완료)

| 필요 기능 | 기존 자산 | 비고 |
|---|---|---|
| 유튜브 대본 → 전략 조건 해석 | `core/nl_strategy.py::interpret_strategy_text()` / `_interpret_staged_strategy_text()` | 그대로 재사용, 대상 종목만 100개로 확장 |
| 단일 종목 백테스트 | `core/backtest_engine.py::run_backtest(ticker, indicator_config, start, end)` | 100종목 루프 + 파라미터 그리드 루프로 감싸면 됨 |
| **3-way 비교 (요청 5번과 정확히 일치)** | `core/backtest_engine.py::compare_with_benchmarks()` — 전략 적용 vs 개별종목 매수보유 vs S&P500 매수보유 반환 | 신규 구현 불필요, 100종목 루프에 그대로 사용 |
| S&P500 종목+섹터+PER/PBR/시총 | `core/screener.py::get_universe()`, `get_fundamentals()` | 100종목 표본 + 가치주/성장주 판별 입력값으로 사용 |
| 성장률 등 밸류에이션 입력값 | `core/valuation.py::fetch_valuation_inputs()` (`earningsGrowth`, `dividendRate` 등 이미 조회) | 성장주 판별에 재사용 가능 |
| 매크로 국면별 유리 섹터 | `core/macro_cycle.py::SECTOR_ROTATION`(4국면 × 섹터 리스트) | 경기민감/방어 섹터 태깅에 재사용 |
| 전략 저장 | `core/strategy_library.py` | 튜닝된 최종 전략도 기존 라이브러리에 저장 → 관심종목 모니터링(모듈 C)과 자연 연결 |

**결론**: 새로 만들 것은 (a) 다종목 배치 실행 루프, (b) 파라미터 그리드 탐색 + train/test 분리
검증, (c) 종목 스타일 판별기, (d) 스타일→탐색방향 매핑 표, (e) 결과 랭킹/선택 UI 5가지뿐이고
백테스트 엔진 자체나 3-way 비교 로직은 손댈 필요가 없음.

## 3. 확정된 설계 결정 (대화로 결정됨)

| 항목 | 결정 |
|---|---|
| 튜닝 방식 | Grid/Random Search + Train/Test 기간 분리 검증 (과최적화 방지). RL은 쓰지 않음. 이후 데이터가 쌓이면 주기적으로 재실행할 수 있도록 설계 |
| 유형별 대응 방식 | 전략 **구조는 원본 그대로 유지**, 종목 유형에 따라 **파라미터 탐색 범위/방향만** 차등 (예: 주도주는 짧은 이평·타이트 손절 위주 탐색, 방어주는 낮은 매매빈도 위주 탐색) |
| Top-N 랭킹 기준 | 기본값 = **초과수익**(전략 CAGR − S&P500 매수보유 CAGR), 사용자가 정렬 기준 변경 + 종목 직접 선택도 가능해야 함 |
| 결과 비교 화면 | 기존 `compare_with_benchmarks()` 3-way 비교를 종목별로 그대로 사용 |
| UI 위치 | 새 페이지 아님. 기존 `app/pages/1_백테스팅.py`(모듈 A) 안에 새 탭으로 추가 |

## 4. 종목 스타일 분류 체계 (확정)

"성장주/주도주뿐 아니라 알고 있는 범위에서 합리적인 분류 체계를 설계해달라"는 요청에 따라, **기존
데이터 소스만으로 계산 가능한** 6개 카테고리를 제안합니다. 카테고리는 상호 배타적이지 않을 수
있어(예: 고성장 반도체 주도주), 종목마다 6개 점수를 모두 계산하고 **가장 점수가 높은 유형을 주
유형(primary type)으로 태깅**하는 방식을 제안합니다.

| 카테고리 | 판별 신호 (데이터 소스) | 대응 전략 탐색 방향(제안) |
|---|---|---|
| **주도주** (모멘텀 리더) | 최근 3~6개월 수익률이 100종목(또는 같은 섹터) 대비 상대강도 상위 percentile + 거래량 20일 평균 우상향 (신규 지표 필요, 가격이력만으로 계산) | 짧은 홀딩, 신고가 돌파형 파라미터(짧은 이평, RSI 상단 돌파), 타이트한 트레일링 손절 위주 탐색 |
| **성장주** | PER이 섹터 평균 대비 프리미엄 + `earningsGrowth` 양호 (`valuation.py::fetch_valuation_inputs()` 재사용) | 추세추종 눌림목 매수, 장기 이평(예: 60~200일) 위 유지 필터, 긴 보유기간 위주 탐색 |
| **가치주** | PER/PBR이 섹터 평균 대비 낮음 (`screener.py::get_fundamentals()` 재사용) | 평균회귀형(과매도 반등), RSI 저점·밴드 하단 반등 파라미터 위주 탐색 |
| **경기민감주** (씨클리컬) | GICS 섹터가 산업재/소재/자유소비재/금융/에너지 등 (`macro_cycle.py::SECTOR_ROTATION`에서 회복·확장·둔화 국면에 반복 등장하는 섹터로 태깅) | 매크로 국면 필터를 진입 조건에 결합(확장기 위주 진입), 국면 전환형 청산 파라미터 탐색 |
| **경기방어주** | GICS 섹터가 유틸리티/헬스케어/필수소비재/통신 (`macro_cycle.py::SECTOR_ROTATION`의 수축 국면 섹터) | 낮은 변동성 활용, 매매 빈도를 낮추는 방향(느슨한 진입/청산 임계값) 위주 탐색 |
| **퀄리티 컴파운더** (우량 장기 우상향주) | 장기(3~5년) MDD가 작고 200일선 위 체류 비율이 높음(신규 지표 필요, 가격이력만으로 계산) | 조정 시 매수(buy-the-dip), 넓은 손절 라인(추세 훼손 전까지 보유) 위주 탐색 |

**신규로 계산해야 하는 지표 2개** (기존 `core/indicators.py`에 없음, 가격이력만으로 계산 가능하므로
외부 데이터 추가 불필요):
- 상대강도(RS) 백분위 순위 — N개월 수익률을 100종목 유니버스 내에서 percentile rank
- 200일선 위 체류 비율 / MDD 기반 추세 일관성 스코어

경기민감주/방어주는 매크로 대시보드(모듈 G)의 국면 판단과 **연동 확정**. 단, `determine_cycle_phase()`는
FRED GDP/실업률 데이터가 필요한데 **FRED_API_KEY가 현재 미설정 상태**(PROGRESS.md 기존 known issue) —
키가 없어도 섹터 기반 씨클리컬/방어 태깅 자체는 항상 동작하고(섹터 정보는 FRED 무관), "지금이 확장기라
씨클리컬 종목 진입 비중을 높인다" 같은 **국면 필터 결합 부분만** 키가 없으면 비활성/안내 문구로
대체(기존 모듈들의 "키 없어도 안 죽는다" 관례 그대로 따름).

## 5. 확정된 기술적 결정 (제 판단, 2026-07-14)

사용자가 "너가 알아서 판단해"라고 위임한 항목들. 아래처럼 결정하고 이유를 남김.

| 항목 | 결정 | 이유 |
|---|---|---|
| 100종목 표본 방식 | **11개 GICS 섹터별 균등 배분**(섹터당 시가총액 상위 ~9개, 총 ~100개), 시총 상위 100 전체를 쓰지 않음 | 스타일 분류·씨클리컬/방어 판별이 섹터별 표본을 전제로 함 — 시총 상위로만 뽑으면 빅테크 편중으로 섹터별 비교가 무의미해짐 |
| Train/Test 분리 | **75% train(과거) / 25% test(최근)**, 시계열 그대로 앞/뒤 분리(랜덤 셔플 금지) | 표준적인 시계열 백테스트 관행. 그리드서치는 train에서만 수행, 최종 채택 파라미터의 정직한 성과는 test(out-of-sample)로 보고 |
| 탐색 방식 | **적응형**: 원본 전략의 튜닝 가능한 수치 파라미터(이평 기간/RSI 임계값/밴드폭 등) 조합 수가 60개 이하면 전수 그리드, 넘으면 60개(기본값, UI에서 "빠름 20 / 보통 60 / 정밀 150"으로 조절 가능) 랜덤 샘플링 | 파라미터 개수가 전략마다 다른데(원본 구조를 유지하므로) 무조건 전수 그리드면 파라미터 3~4개만 돼도 조합이 폭발 — 예산 상한을 두는 랜덤서치가 100종목 규모에서 현실적 |
| 목적함수 | Train 기간 **샤프지수 최대화**, 단 매매 횟수가 너무 적은(예: 5회 미만) 조합은 후보에서 제외 + `diagnose_strategy_health` 재사용해 진입=청산 자기모순 같은 병적인 조합도 자동 배제 | 단순 수익률 극대화는 소수의 큰 승리에 과최적화되기 쉬움. 위험조정수익 + 기존 안전장치 재사용 |
| 스타일별 탐색 방향 반영 방식 | 원본 파라미터 값을 중심으로 스타일에 따라 탐색 범위(하한/상한)만 이동 — 예: 주도주는 이평 기간 범위를 원본의 0.3~0.8배(짧게), 성장주는 0.8~2배(길게)로 설정. 조건 로직/지표 종류는 그대로(백본 유지 원칙 재확인) | 3절에서 이미 "구조 유지, 탐색 방향만 차등"으로 확정된 원칙의 구체적 구현 규칙 |
| 결과 화면 기준 기간 | 3-way 비교(전략 vs 종목 홀딩 vs S&P500 홀딩)는 **test 구간 기준**을 기본으로 표시, "전체 기간으로 보기" 토글 제공 | test가 정직한(미래 예측에 가까운) 성과. 전체 기간은 참고용으로만 |

## 6. 장기 운영 설계 (반영구 재튜닝 전제)

사용자가 "완성되면 평생 쓰면서 6개월마다 재튜닝할 계획"이라고 밝힘 — 1회성 최적화 도구가 아니라
**이력이 누적되는 시스템**으로 설계해야 함:

- 매 튜닝 실행을 **하나의 배치 레코드**로 DB에 영구 저장(파라미터/성과/사용한 종목 표본/train-test
  구간을 통째로 스냅샷). 같은 원본 전략에 대해 6개월마다 재실행하면 자연히 **회차별 이력**이 쌓임
  (예: 2026-07 1차 튜닝, 2027-01 2차 튜닝, ...).
  - 각 재튜닝 시점마다 test 구간이 최근 쪽으로 이동하므로(예: 데이터가 6개월 더 쌓인 상태에서 다시
    75/25 분리), 이 자체가 일종의 **자연스러운 walk-forward 검증**이 됨 — 매번 복잡한 다중 폴드
    walk-forward를 구현하지 않아도, "반년마다 재실행"이라는 사용 패턴 자체가 장기적으로 walk-forward와
    같은 효과를 냄. (엔지니어링 단순성과 견고성을 동시에 얻는 절충안)
  - 같은 종목·같은 원본 전략의 회차 간 파라미터 변화를 나란히 비교하는 화면은 이번 1단계 범위에는
    넣지 않고, 이력이 2회 이상 쌓인 뒤 필요성을 보고 추가할지 결정(과설계 방지).
- **자동 스케줄링은 이번 단계에서 만들지 않음** — 재튜닝은 계산량이 크고(100종목 × 파라미터 탐색)
  결과를 사람이 검토 후 전략 라이브러리에 반영할지 판단하는 게 안전하므로, 기존 "주간 인사이트
  리포트 생성" 버튼과 같은 패턴으로 **수동 트리거**(버튼)로 시작. 나중에 반기 자동 스케줄이
  필요해지면 `scheduler/run_scheduler.py`에 잡을 추가하는 건 쉬움(기존 job_manager 백그라운드 실행
  패턴 재사용).

## 9. 직접 수식 전략 전용 Gemini 기반 튜닝 (2026-07-14, 3/5절 원칙의 명시적 예외)

**중요**: 이 절의 내용은 **직접 수식(expression) 전략에만 적용**된다. JSON(레짐/1:2:6) 전략의
"백본 유지, 숫자만 튜닝, RL 없음" 원칙(3/5절)은 전혀 바뀌지 않았다 — `build_param_grid()`가
지표 종류/조건 개수/AND-OR 로직을 절대 안 바꾸는 동작은 그대로다.

직접 수식 전략은 숫자 파라미터가 JSON 필드명이 아니라 자유 문자열 안에 있어 원래 튜닝 대상을
구조적으로 식별할 수 없었다(7절 "알려진 한계"에 명시했던 내용). 사용자가 이 한계를 Gemini로
넘어달라고 요청해 2단계로 구현했다(`core/strategy_tuning.py`):

| 단계 | 함수 | 동작 |
|---|---|---|
| 1. 숫자 식별 | `identify_tunable_numbers(expression)` | 숫자 리터럴의 실제 값/위치는 `ast`로 결정론적으로 추출(Gemini가 값을 잘못 베껴 써서 수식이 깨질 위험 원천 차단), Gemini에게는 "이 숫자가 튜닝 가능한가/역할/합리적 범위"라는 의미 판단만 맡김. 응답 개수가 숫자 개수와 안 맞으면 신뢰 불가로 보고 빈 리스트(원본 폴백). |
| 1. 숫자 튜닝 | `_build_expression_param_grid()` | 식별된 숫자만 스타일 방향(기존 배수 범위)과 Gemini 제안 범위를 함께 반영해 후보 생성, `validate_syntax()`로 치환 결과가 실행 가능한지 매번 검증 후 통과한 것만 후보에 포함. 이후 `tune_strategy_for_ticker()`가 기존 JSON 전략과 완전히 동일한 절차(train 샤프 최대화 + 자기모순/거래횟수 필터 + test 검증)로 처리한다. |
| 2. 구조 변경(escape hatch) | `generate_structural_variants()` + `tune_expression_strategy_for_ticker()` | 1단계 결과가 test 구간에서 종목 매수보유 + S&P500 매수보유를 **둘 다** 못 이길 때만 호출. Gemini에게 원본과 다른 지표 구성의 대안 수식을 최대 3개 제안받고(문법은 `core/expression_engine.py`의 허용 목록만 사용하도록 프롬프트에 명시), 각각 `validate_syntax()`로 걸러낸 뒤 1단계와 동일한 절차로 재검증. 전체 후보(원본 숫자튜닝 결과 + 구조변형 후보들) 중 test 전략 CAGR이 가장 높은 것을 채택. |

**설계 판단(사용자가 위임, 근거)**:
- **반복 진화 대신 1회성 후보 생성**: 구조 탐색은 파라미터 탐색보다 과최적화 위험이 훨씬 크고
  (탐색 공간이 사실상 무한), 여러 세대에 걸친 피드백 루프는 API 호출량도 선형으로 늘어난다. 1회에
  최대 3개만 받아 기존과 동일한 train/test 절차로 검증하면 숫자 튜닝과 같은 수준의 통제된 위험을
  유지하면서도 탐색 범위를 넓힐 수 있다.
- **test에서 여전히 못 이겨도 계속 재시도하지 않고 최선으로 폴백**: `tune_strategy_for_ticker()`의
  기존 "유효 후보 없으면 원본" 관례와 일관되게, 구조 변경 후보가 전부 기존 결과보다 나쁘면 절대
  채택하지 않는다(손해 보는 방향으로 안 감). 결과에 `outperformed_ticker_bh`/
  `outperformed_benchmark_bh`(bool) 플래그를 항상 포함시켜, 실제로 이기지 못했으면 결과가 조용히
  성공으로 보이지 않게 한다(UI 결과 표의 CAGR 컬럼들로 직접 비교 가능).
- **DB**: `StrategyTuningResult`에 `backbone_changed`(Boolean) 컬럼 추가(구조가 실제로 바뀌었는지
  추적). 기존 `data/quant.db`에 이미 이 테이블 행이 있어(1회 실행분) 수동 `ALTER TABLE`로 마이그레이션.
- **검증**: `tests/test_strategy_tuning.py`에 단위테스트 15개 추가(숫자 추출/식별/치환, 구조 변형
  후보 필터링, 1단계로 충분하면 2단계 스킵, 2단계가 1단계보다 항상 우위이거나 최소 동등, 배치 실행
  dispatch) — 전부 Gemini 호출을 mock. 그리고 실제 Gemini API + 실제 AAPL 데이터로 전체 파이프라인을
  라이브 실행해 확인: `identify_tunable_numbers`가 20/14/30을 각각 "이동평균 기간/RSI 계산 기간/RSI
  과매도 임계값"으로 정확히 역할 판별, `generate_structural_variants`가 실행 가능한 대안 2개 생성,
  `tune_expression_strategy_for_ticker`를 AAPL(2021~2026, 강세장 구간)에 돌려보니 구조 변경까지
  갔는데도 여전히 매수보유를 못 이겼고 — 이때 `outperformed_ticker_bh=False`가 그대로 정직하게
  보고됨을 확인(과최적화로 억지로 "이겼다"고 보이게 만들지 않음, 의도한 안전장치가 실제로 작동).

## 10. 튜닝 방식 변경 — 종목별 독립 탐색 → 스타일 그룹 풀링 트레이닝 (2026-07-15, 사용자 확정)

사용자 요청 원문: "트레이닝을 할 때 경기방어주, 주도주 등으로 우선 데이터셋을 나눈 후 각자의 전략은
각 데이터셋에 대해서만 트레이닝을 진행했으면 해. 그리고 어떻게 해서든 S&P500 홀딩은 이기게 하며
각 개별 종목들도 바이 앤 홀드보다 더 나은 값을 가질 수 있도록 알고리즘을 개선해줘."

두 가지 요청이 섞여 있어 코드 작성 전 `AskUserQuestion`으로 확인했다:

1. **그룹 단위 트레이닝**(명확, 바로 구현): 종목마다 따로 탐색하던 기존 방식을 버리고, 같은 스타일
   (4절의 6개 카테고리)로 묶은 종목 전체를 하나의 데이터셋으로 써서 그 그룹에 공통으로 잘 맞는
   설정 하나를 찾는다.
2. **"어떻게 해서든 이기게"**(모호 — train/test 분리라는 기존 확정 원칙과 충돌할 여지가 있어 확인
   필요): test(out-of-sample) 구간 성과를 후보 선택 기준에 반영하면 사실상 답을 보고 답을 고르는
   것(데이터 스누핑)이 되어, 이 백테스트에서는 항상 이긴 것처럼 보이지만 실제 미래 성과와는 무관해질
   위험을 설명하고 두 가지를 물어봄:
   - **Test 구간에서도 못 이기는 종목/그룹 처리**: "탐색은 최대한 넓히되 정직하게"(권장, 채택) vs
     "test 성과까지 선택 기준에 반영해 반드시 이기게"(데이터 스누핑 위험 — 채택 안 함).
   - **구조 변경 escape hatch(9절, 기존엔 직접 수식 전략에만 있었음) 확장 여부**: "전체 전략
     유형(레짐/1:2:6 포함)으로 확장"(권장, 채택) vs "직접 수식에만 유지".

### 구현 (`core/strategy_tuning.py`)

| 항목 | 내용 |
|---|---|
| 그룹 구성 | `run_batch_tuning()`이 `compute_style_scores()`로 종목별 style_type을 먼저 계산하고, 같은 style_type끼리 묶어 그룹별로 `tune_strategy_for_group()`을 한 번씩만 호출(종목별 반복 호출 제거). 같은 그룹의 종목들은 결과적으로 `tuned_config`가 서로 동일해진다. |
| Train(그룹 풀링) | `_select_best_group_config()`: 후보마다 그룹에 속한 **모든** 종목의 train 구간에서 샤프지수를 구해 평균을 목적함수로 삼는다. 한 종목에만 우연히 맞는(과최적화된) 후보를 그룹 대표로 뽑지 않기 위해, 그룹의 최소 50%(`_MIN_GROUP_COVERAGE_RATIO`) 이상 종목에서 유효한 결과(매매 5회 이상)가 나와야만 후보로 인정한다(`_candidate_group_train_sharpe`). |
| Test(정직성 유지) | `_evaluate_group_config_on_test()`: 확정된 그룹 config를 종목별로 test 구간에서 **개별** 평가만 하고, 이 결과는 절대 후보 선택에 되먹임하지 않는다 — train/test 분리 원칙은 이번 변경에서도 어기지 않는다(사용자 확정). |
| Escape hatch 확장 | `generate_structural_variants_for_config()`가 스키마 유형에 따라 위임: 직접 수식은 기존 `generate_structural_variants()`(문자열) 재사용, 레짐/1:2:6은 `core/nl_strategy.py`의 기존 `INDICATOR_CONFIG_SCHEMA`/`STAGED_INDICATOR_CONFIG_SCHEMA`를 그대로 재사용한 JSON 구조 생성으로 대응(새 스키마 중복 정의 없음). 그룹 평균 test 초과수익이 0 이하일 때만 1회성으로 트리거되고(반복 진화 없음, 9절과 동일한 과최적화 통제 원칙), 그룹 평균이 실제로 개선되는 변형만 채택한다. |
| 정직한 실패 보고 | 그룹 평균이 이 escape hatch를 거치고도 여전히 못 이기면 그대로 낮은 초과수익/CAGR로 보고된다(`outperformed_*` 같은 별도 은폐 플래그 없이 UI가 CAGR 숫자를 그대로 보여줌 — 승률을 인위적으로 끌어올리지 않는다는 확정 원칙이 그대로 드러남). |
| 삭제된 함수 | `tune_expression_strategy_for_ticker()`(종목 단위 escape hatch 진입점)는 `tune_strategy_for_group()`에 역할이 흡수되어 제거. `tune_strategy_for_ticker()`/`generate_structural_variants()`/`identify_tunable_numbers()` 등 9절의 나머지 빌딩 블록은 그대로 유지되고, "🔬 알고리즘 자동 생성" 탭(`generate_and_backtest_strategy`, 종목 단위 백지생성 — 이번 변경의 대상이 아님)이 여전히 `tune_strategy_for_ticker()`를 그대로 사용한다. |

### UI (`app/pages/1_백테스팅.py`)

"🧬 다종목 미세튜닝" 탭 상단 설명을 그룹 풀링 방식으로 갱신하고, 결과 표 위에 **"스타일 그룹별
요약"** 표를 추가(종목수/평균초과수익/승률/백본변경 — 이미 저장된 종목별 결과에서 즉석 집계라

### 검증

`tests/test_strategy_tuning.py`에 그룹 풀링 관련 신규 유닛테스트 다수 추가(그룹 커버리지 최소 조건,
그룹 평균 샤프/초과수익 계산, 그룹 전체가 동일 config를 공유하는지, test 구간이 선택에 전혀 영향을

진행). 전체 `pytest` 310개 통과. 실제 Gemini API + 실제 가격 데이터(AAPL/MSFT/JNJ)로 `run_and_save_
tuning()` 전체 파이프라인을 라이브 실행해, 스타일 분류 → 그룹별 풀링 튜닝 → DB 저장/조회까지 정상
동작하고 같은 스타일 종목들이 실제로 동일한 tuned_config를 공유함을 확인(검증용 실행 결과는 이후 삭제).
`AppTest`로 기존(이 변경 이전에 저장된, 종목별 독립 탐색 방식) 실행 이력을 불러와도 그룹 요약 표가
예외 없이 렌더링됨을 확인(과거 데이터 호환성).

## 11. 파라미터 업데이트 체계화 — 다중 구간 워크포워드 + 안정성 점수 + 튜닝 리포트 (2026-07-15)

### 11.1 요청 배경

사용자가 "튜닝을 했는데도 실제 과거 데이터 기반인데 왜 계속 시장을 못 이기는지 이해가 안 된다,
극단적으로 overfitting하면 이길 텐데"라는 질문을 함. 대화로 다음을 확인/설명함:

- **overfitting은 실제로 train에서는 이긴다** — 문제는 이 엔진이 이미 train/test를 분리해 test
  성과만 정직하게 보고하도록 설계돼 있다는 것(4b절). "시장을 못 이긴다"고 보이는 결과는 test
  (out-of-sample) 구간이고, 이건 버그가 아니라 overfitting 적발 장치가 정상 작동하는 신호.
- test 기준까지 선택에 반영해 "무조건 이기게" 만드는 것은 데이터 스누핑이라 이미 10절에서
  기각된 방향 — 이번에도 동일 원칙 재확인, 절대 test를 선택 기준에 섞지 않음.
- 강세장 개별종목 매수보유는 가격/지표 기반 타이밍 전략이 구조적으로 이기기 어려운 벤치마크
  (9절 AAPL 사례: 구조 변경까지 시도했는데도 강세장에서 매수보유를 못 이김 — 이번 절에서 동일
  패턴이 재현됨, 아래 11.3 참고).

사용자가 "파라미터 업데이트를 더 체계적으로" 하고 싶다고 요청 → grid search 자체를 더 견고하게
만드는 방향(다중 구간 워크포워드/평평한 영역 선호/다중검정 보정 등)을 논의 후, **다중 구간
워크포워드 + 안정성(평평한 영역) 선호 조합**을 제안하고 실제 AAPL 데이터로 데모 실행 → 결과를
함께 확인한 뒤 "문서 정리 → 구현 → 이후 백본 구조 한계 논의" 순서로 진행하기로 확정.

### 11.2 결정: 단일 train 구간 평가 → 다중 폴드 평가 + 안정성 점수

기존 `_select_best_group_config`는 train 구간(`train_start~train_end`) 전체를 하나로 보고 그룹
평균 샤프지수가 가장 높은 후보를 뽑았다. 이를 아래처럼 바꾼다(4b절의 train/test 분리 원칙은
전혀 바뀌지 않음 — test 구간은 여전히 선택에 관여하지 않고 최종 검증 전용):

1. train 구간을 `_WALK_FORWARD_FOLDS`(기본 3)개의 연속된 하위 구간(폴드)으로 균등 분할한다.
2. 각 후보를 폴드마다 독립적으로 평가한다 — 기존 `_candidate_group_train_sharpe(tickers, candidate,
   start, end)`가 이미 임의의 날짜 구간을 받게 설계돼 있어, 폴드 날짜만 넘기면 그대로 재사용 가능
   (신규 백테스트 로직 불필요).
3. 후보 점수 = 폴드 간 평균 샤프지수 − `_ROBUSTNESS_PENALTY_WEIGHT`(0.5) × 폴드 간 표준편차.
   특정 폴드(특정 시기)에만 우연히 잘 맞고 나머지 폴드에서는 나쁜 후보(뾰족한 피크 = 전형적
   overfitting 신호)에 패널티를 줘서, 여러 시기에 걸쳐 일관되게 괜찮은("평평한 고원") 후보를
   우선한다.
4. 폴드 커버리지도 그룹 커버리지(`_MIN_GROUP_COVERAGE_RATIO`)와 같은 방식으로 최소 절반 이상
   폴드에서 유효한 결과(매매 5회 이상)가 나와야 후보로 인정한다.
5. 확정된 그룹 config는 기존과 동일하게 test(마지막 25%, out-of-sample) 구간에서 종목별 개별
   평가만 하고 선택에는 절대 되먹임하지 않는다(4b절 원칙 그대로).

### 11.3 실증 데모 결과 (중요 — 방법론의 한계를 정직하게 기록)

구현 전, AAPL 실제 가격 데이터(2018~2026)로 프로토타입 스크립트를 돌려 방법론을 먼저 검증함
(백본: MA20/60 골든크로스 AND RSI(14)<30 매수). 3개 폴드(2021-07~2022-06 / 2022-07~2023-06 /
2023-07~2024-06)에서 평가해 "MA16/120, RSI(14)<40"을 선택(폴드 평균 샤프 +1.25, 표준편차 0.02 —
원본보다 훨씬 안정적). 그런데 선택 과정에서 단 한 번도 보지 않은 최종 홀드아웃(2024-07~2026-06,
조정 없는 강세장 구간)에서 검증한 결과:

| | 전략 CAGR | 거래 | AAPL 매수보유 | S&P500 매수보유 |
|---|---|---|---|---|
| 원본 백본 | -0.14% | 2회 | +14.06% | +16.64% |
| 워크포워드 튜닝 결과 | -1.41% | 9회 | +14.06% | +16.64% |

**결론**: 다중 폴드 검증에서 "훨씬 안정적"으로 나온 후보가 실제 미래 구간에서는 원본보다도
나빴다. 이는 방법론 결함이 아니라, 9절 결론의 재확인이다 — 조정 없이 밀어 올리는 강세장에서는
"쉬었다 들어가는" 모든 타이밍 전략이 매수보유보다 구조적으로 불리하고, 과매도(RSI) 진입 조건은
이런 국면에서 신호 자체가 잘 안 뜨거나(원본: 2회 매매) 무리하게 조건을 완화하면 오히려 나쁜
타이밍에 매매를 늘리게 된다(튜닝본: 9회 매매, 결과는 더 나쁨). **다중 구간 워크포워드 +
안정성 점수는 "더 정직하고 견고한 검증 방법"이지 "시장을 이기게 해주는 방법"이 아니다** — 사용자도
이 한계를 이해하고 채택함. 백본 자체의 구조적 한계(AND 결합, 과매도 전용 진입 등)를 다루는 논의는
다음 대화로 이어감(별도 절 예정).

expander가 실제 실행 결과(볼린저/MA+캔들 구조 변경 포함)를 예외 없이 렌더링하는지 확인(검증용
실행 기록은 확인 후 DB에서 삭제).

## 12. 백본 구조 한계 대응 — 결정론적 국면 스위치 변형 (직접 수식 전용, 2026-07-15)

### 12.1 배경 — 지금까지 두 데모에서 확인된 구조적 원인

11.3절 AAPL 데모와 9절 AAPL 사례에서 공통으로 드러난 문제를 세 가지로 정리:

1. **AND 결합이 신호를 희소하게 만든다** — "골든크로스(추세 시작) AND RSI 과매도(조정)"는 논리적으로
   거의 동시에 안 일어남(추세 시작 시점엔 RSI가 보통 상승 중). 11.3절 원본이 8년에 2번만 거래한 이유.
2. **청산이 대칭적으로 너무 이르다** — 진입(과매도) 조건이 깨지자마자 청산이라 반등 초반 몇 %만 먹고
   나옴. 추세가 지속돼도 못 올라탐.
3. **국면 무감각** — 전체 기간에 동일한 조건 하나(평균회귀형)를 그대로 적용. 조정 없이 밀어 올리는
   강세장에서는 "조정 매수" 로직 자체가 원천적으로 안 맞는데, 이를 감지하고 다르게 행동할 방법이 없음.

1/2번은 3번(국면 무감각)의 파생 증상 — 조정이 없는 국면에서는 평균회귀 진입 자체가 구조적으로
안 맞다는 하나의 원인에서 나온다. 사용자와 논의 후, 종목 자체의 추세(시장 전체 국면이 아니라 이
티커의 가격이 실제로 어떻게 움직였는지)를 기준으로 진입 로직 자체를 스위치하는 방향으로 확정.

### 12.2 핵심 발견 — 엔진 확장 없이 이미 표현 가능

`core/expression_engine.py`에 이미 `sma`(이동평균)/`highest`(N일 롤링 최고가)/`rsi` 등이 있어,
국면 스위치는 새 지표 없이 기존 함수만으로 한 줄의 불리언 수식으로 표현된다:

```
(close > sma(close, 200) and close >= highest(close, 60))    # 종목 자체 상승추세: 신고가 돌파(추세추종)
or
(close <= sma(close, 200) and (원본 진입 조건))                # 그 외: 원본 진입 조건 그대로
```

문제는 이 "(A and B) or (C and D)" 형태의 중첩 논리가 **직접 수식(expression) 스키마에서만
표현 가능**하다는 것 — 레짐/1:2:6(JSON) 스키마는 조건이 flat AND/OR 리스트라 애초에 중첩을 표현할
방법이 없고(engulfing 같은 캔들패턴 조건은 수식 문법에 아예 없음), 그러려면 스키마 자체를 갈아엎어야
한다. 9절에서 이미 한 번 "직접 수식에만 적용, JSON 백본 유지 원칙은 그대로"로 정리했던 것과 같은
갈림길이라, 이번에도 동일하게 **직접 수식 전략에만 적용**하기로 확정(레짐/1:2:6은 기존 Gemini
구조변경 escape hatch를 그대로 유지 — 범위를 넓히지 않음, 과설계 방지).

### 12.3 결정된 설계

| 항목 | 결정 |
|---|---|
| 생성 방식 | **결정론적 고정 템플릿**, Gemini 창의성에 안 맡김 — 추세 필터=`close > sma(close, 200)`, 추세추종 진입=`close >= highest(close, 60)`(60일 신고가 돌파). 매번 다른 제안이 나오는 기존 구조변경 escape hatch(9/10절)와 달리 재현 가능 |
| 숫자(200/60) 튜닝 | 생성 후 기존 `_build_expression_param_grid`(3b절, `identify_tunable_numbers`로 Gemini가 의미 판단)가 원본 진입 조건의 숫자들과 함께 그대로 스타일별로 조정 — 새 튜닝 경로 불필요 |
| 통합 지점 | 새 트리거/새 흐름을 만들지 않고, **기존 구조변경 escape hatch가 이미 트리거되는 시점**(그룹 평균이 test에서 못 이길 때, 4b/9절)에 Gemini가 제안하는 변형들(최대 3개) 옆에 이 결정론적 후보를 하나 추가. 나머지 선택 절차(walk-forward 점수 → test 검증 → 개선될 때만 채택)는 기존과 완전히 동일 |
| 실패 시 | 결합 결과가 문법상 실행 불가능하면(원본 진입 조건 자체가 이미 유효했다면 사실상 발생하지 않지만 방어적으로) `validate_syntax()`로 걸러 None 반환 → 후보에서 조용히 제외(기존 "유효 후보 없으면 원본 유지" 관례와 동일) |

## 13. 국면별(약세장/강세장) 분리 트레이닝 — 야간 미세튜닝 엔진 확장 (2026-07-16)

### 13.1 요청

사용자 요청 원문: "야간에 테스팅할 때 S&P500을 기준으로 약세장/강세장을 나누고 약세장에서의 데이터를
가지고 트레이닝하는 전략/강세장에서의 트레이닝하는 전략을 따로따로 접근해서 미세튜닝하도록 변경해줘".

"야간에 테스팅"은 정확히 `scheduler/run_scheduler.py::strategy_nightly_tuning_job()`(매일
00:05~04:00 KST, `core.strategy_tuning.run_and_save_tuning()`을 반복 호출)을 가리킨다. 이 잡은
`run_batch_tuning()` → `tune_strategy_for_group()`을 그대로 호출하는 얇은 래퍼이고, "🧬 다종목
미세튜닝" 탭(수동 실행)도 같은 파이프라인을 쓰므로, 엔진(core 함수) 레벨에서 바꾸면 별도 토글 없이
두 진입점(야간 자동 + 수동 UI) 모두에 자동 반영된다 — 이번 변경의 적용 지점.

### 13.2 핵심 설계 — 스타일 그룹 안에 "국면" 축을 추가

기존(10/11절): 스타일(6개 카테고리) → 그룹당 공통 config 1개, train 구간 전체를 3개 **달력 기준**
폴드로 나눠 워크포워드 평가.

변경 후: 스타일 × 국면(약세장/강세장, 2종) → 그룹당 config **2개**. train 구간을 달력 기준 등분
대신, 그 구간 안에서 실제로 약세장이었던 날들만 모은 폴드로 한 번, 강세장이었던 날들만 모은 폴드로
또 한 번 — 완전히 독립적으로 워크포워드 탐색을 돌린다. "중립/혼조" 국면 날짜는 양쪽 트레이닝 어디에도
쓰지 않는다(애매한 날을 억지로 한쪽에 욱여넣지 않기 위함).

### 13.3 국면 판정 방법 (기술적 결정)

`core/market_regime.py::get_market_regime_snapshot()`의 4-신호 스코어(추세/골든크로스/낙폭/시장폭)는
"현재 시점 1건"을 위해 설계됐다 — 특히 시장폭(breadth)은 S&P500 전종목의 200일선을 매번 다시
조회해야 해서, 5년치 이력의 날짜마다 반복 계산하기엔(야간마다 재실행됨) 비용이 지나치게 크다.

대신 **벤치마크(^GSPC) 종가 하나의 시계열만으로 pandas 벡터 연산으로 한 번에 계산 가능한 2개 신호**만
재사용해 일별로 결정론적 라벨을 매긴다 (`core/market_regime.py` 신규 함수):
- 200일선 대비 위치(기존 `score_trend_position` 로직과 동일 신호)
- 52주 고점 대비 낙폭 (기존 `BEAR_MARKET_DRAWDOWN_PCT = -20%` 상수 재사용 — "공식 약세장" 정의)

라벨 규칙: 낙폭이 -20% 이하면 무조건 "약세장"(다른 신호와 무관하게). 그렇지 않고 종가가 200일선 위 +
낙폭이 -10% 이내(뚜렷한 조정이 없는 상태)면 "강세장". 나머지(200일선 아래인데 -20%까지는 아니거나,
200일선 위인데 -10~-20% 조정 중)는 "중립"으로 남겨 트레이닝 대상에서 제외한다.

### 13.4 연속 구간(세그먼트) 추출 — 기존 워크포워드 폴드 인프라 그대로 재사용

일별 라벨을 국면별로 묶어 연속 날짜 구간(세그먼트) 리스트로 변환한다(예: 2022년 약세장 전체가 구간
하나, 2020년 3월 코로나 급락이 또 다른 구간 하나). 10 거래일 미만인 짧은 구간(경계 근처 flicker)은
버린다 — 다른 국면에 편입시키지 않고 그냥 트레이닝에서 제외한다. (실제 2018~2023 데이터로 검증하며
처음엔 20 거래일로 잡았다가, 2020년 코로나 급락처럼 -20% 이상 낙폭이 지속된 기간이 매우 짧은(약
17거래일) 극단적으로 빠른 약세장까지 flicker로 오인해 통째로 버리는 것을 발견해 10으로 낮췄다 —
회복이 빨라도 실제 약세장이었던 구간을 놓치면 안 되므로 보수적으로 조정.)

이 세그먼트 리스트를 기존 `_select_best_group_config_walkforward(tickers, candidates, folds)`의
`folds` 인자로 그대로 넘긴다 — 폴드 간 평균-표준편차 점수 계산, 최소 커버리지 요구, 트레일 기록 등
11절에서 만든 로직을 전혀 새로 만들 필요가 없다. 유일한 변경은 폴드를 "달력 등분"
(`_split_into_folds`) 대신 "실제 국면 구간"으로 만드는 새 함수 하나뿐이다. 약세장 구간이 강세장보다
훨씬 드물고 짧다는 점(5년에 보통 1~3개 구간)은 문제가 아니다 — 폴드 개수가 가변적이어도 워크포워드
점수 계산 로직은 그대로 동작한다(오히려 폴드가 실제 사건 단위라 "특정 시기에만 우연히 맞는 후보"를
걸러내는 원래 취지에 더 가깝다).

### 13.5 Train 데이터 부족 시 폴백

5년 lookback 안에 특정 국면 구간이 아예 없거나(예: 조회 구간에 뚜렷한 약세장이 없음) 최소 커버리지
(`_group_min_required`, 그룹 종목의 최소 절반 이상에서 유효 결과)를 못 채우면
`_select_best_group_config_walkforward`가 기존과 동일하게 `None`을 반환하고, `tune_strategy_for_group`
의 기존 폴백(원본 config 그대로 채택)이 그대로 작동한다 — 새 예외 처리 불필요. 다만 조용히 원본과
똑같은 숫자가 나오면 사용자가 "튜닝이 실행됐는데 왜 원본이랑 같지?"로 오해할 수 있어, 결과에
`insufficient_regime_data`(bool) 플래그를 추가해 UI가 그 사실을 명시적으로 안내한다.

### 13.6 Test 평가 방식 — 국면 일치 구간만 평가 (2026-07-17 사용자 정정)

4b/11절에서 이미 확정한 원칙("test 구간은 선택에 절대 관여하지 않는다, 데이터 스누핑 금지")은 이번
변경에서도 전혀 바뀌지 않는다.

**최초 설계(위 문단, 폐기됨)**: test 구간 전체(국면 구분 없이 마지막 25% 전부)를 기본/주 지표로
삼고, 국면이 일치하는 구간만 보는 평가는 `regime_matched_test`라는 보조 지표로만 붙였었다.

**사용자 정정(2026-07-17)**: "약세장의 전략을 강세장에서 테스트할 필요는 없잖아 — 결국 두 개의
전략을 각각 만들어야 하는 것"이라는 피드백을 받고 전면 수정했다. 약세장용으로 학습한 config를
test 구간이 우연히 강세장이었던 부분까지 포함해 평가하면, 애초에 "이 config는 약세장에서 쓸
것"이라는 전제와 안 맞는 성과가 섞여 들어가 오히려 오해를 준다 — 학습(train)뿐 아니라 검증(test)도
같은 국면 데이터로만 해야 "약세장 config가 약세장에서 얼마나 통하는지"를 정직하게 answer할 수 있다.

**확정 방식**: `regime`이 지정된 호출(약세장/강세장 각각)은 test 구간 안에서 **같은 국면의 가장 긴
연속 구간 하나만**으로 평가한다(스티칭 없음 원칙은 유지 — 여러 조각을 이어붙이면 갭에서 포지션이
모호해진다). 이 평가 결과가 이제 `group_mean_excess_return`/`group_win_ratio`/
`per_ticker_test_comparison`(=구 `test_comparison`) 등 **주 지표 자리를 그대로 차지**한다 — 더 이상
국면 무관 전체기간 평가는 계산하지 않는다(중복 계산 제거, 개념적으로도 하나만 존재). 구조 변경
escape hatch(mean_excess≤0일 때 대안 시도)의 트리거 판단도 이제 이 국면 일치 평가를 기준으로 한다.
`regime_matched_test` 필드는 이제 `segment_start`/`segment_end`(어느 구간으로 검증했는지)를 함께
보여주는 용도로 유지하되, 내용상 `per_ticker_test_comparison`과 사실상 같은 데이터를 담는다.
`regime=None`(레거시 달력 폴드 경로)은 이번 정정과 무관 — 기존처럼 test 구간 전체로 평가한다.

**국면 일치 구간이 test 안에 아예 없으면**: 조용히 다른 데이터로 대체하지 않고 `group_mean_excess_
return=None`("검증 불가")으로 정직하게 보고한다. escape hatch도 이 경우엔 대안을 채택하지 않는다
(비교 기준 자체가 없어 "개선됐다"고 판단할 근거가 없으므로) — 없는 데이터를 억지로 만들어내지
않는다는 기존 원칙(9/10절)의 연장.

### 13.7 DB/데이터 모델

`StrategyTuningResult`에 컬럼 3개 추가: `trained_regime`(String, "약세장"|"강세장"),
`insufficient_regime_data`(Boolean), `regime_matched_test`(Text, JSON). 종목 하나당 배치 1회 실행에

`ALTER TABLE`로 마이그레이션 필요.

### 13.9 라이브 국면 판단 → 두 전략 중 하나 선택 (2026-07-17, 사용자 요청으로 범위 확정)

13.8절에서 "이번 요청 범위 밖"이라고 뒀던 것을 사용자가 명시적으로 요청해 범위 안으로 끌어왔다.
사용자 발언: "약세장/강세장 두 전략을 만든 후, 현재 상황(또는 현재 티커)이 두 케이스 중 어디인지
판단해서 둘 중 하나를 택해 알고리즘 매매를 하려는 것"— 즉 이 튜닝 엔진의 최종 산출물은 "전략
라이브러리에 저장된 config 2개"가 아니라 "지금 어느 걸 써야 하는지까지 알려주는 판단 로직"까지
포함해야 한다.

**결정된 설계**:
- `core/market_regime.py::select_regime_for_trading(snapshot=None)` 신설 — "약세장"/"강세장" 둘 중
  하나로 확정해서 돌려준다. `get_latest_market_regime_snapshot()`(매일 00:00 KST
  `market_snapshot_job`이 미리 계산해두는 것, 13.3절의 일별 라벨링과 별개 — 이건 "지금 이 순간"용
  4-신호 종합 점수)이 "중립/혼조"를 반환하면, 억지로 무시하지 않고 `total_score`의 부호로 소프트하게
  기울이되 `is_ambiguous=True`로 표시한다 — 애매한 상태를 애매하다고 솔직히 알리는 것과, 그래도
  둘 중 하나는 선택해야 하는 실전 요구사항 사이의 절충(위임받은 기술 판단, 부호 외 다른 규칙을 쓸
  근거가 없어 가장 단순한 규칙 채택).
- `core/strategy_tuning.py::select_live_strategy(bear_strategy_id, bull_strategy_id)` 신설 — 위
  판단 결과에 따라 전략 라이브러리에서 해당 id의 Strategy를 찾아 config와 함께 반환한다. 둘 중 어느
  strategy_id를 쓸지는 호출부가 정하고(예: "가격+거래량 결합 전략 (약세장/강세장 학습)"의 각 id),
  이 함수는 "지금은 어느 게 맞는지"만 판단한다 — 종목 배정(watchlist)이나 실제 매매 실행까지는
  건드리지 않는다(관심종목 모니터링 모듈 C와의 연동은 두 최종 결합 전략이 실제로 만들어진 뒤 다음
  단계로 진행 — 과설계 방지 원칙 유지, 지금은 판단 기본 로직만).
- **국면 무관 원칙(S&P500 기준)**: 사용자가 "현재 상황 혹은 현재 티커"라고 했지만, 13.1~13.4절에서
  이미 "S&P500 기준"으로 국면을 정의하기로 확정했으므로 개별 티커 단위 국면 판단은 만들지 않는다 —
  종목마다 다른 국면 판정 기준을 두면 국면 정의 자체가 갈라져 트레이닝 때 쓴 기준과 실전 판단 기준이
  달라지는 위험이 생긴다(트레이닝-실전 일관성이 더 중요하다고 판단, 위임받은 기술 판단).
  **→ 14절에서 사용자가 이 원칙 자체를 뒤집는 새 요청을 해 별도 축으로 추가됨(13절은 그대로 유지).**

### 13.10 약세장/강세장 2분류 → 약세장/강세장/횡보장 3분류로 확장 (2026-07-17, 같은 날 후속 요청)

사용자가 "백테스팅 탭 미세튜닝이든 야간 미세튜닝이든 강세장/약세장/횡보장으로 데이터셋을 나눠 각각
전략을 만들어라"고 요청. 13.3절까지는 "200일선 위/아래 + 조정폭"만으로 판정해 애매한 날("중립")을
트레이닝에서 통째로 버렸는데, 이 "중립" 잔여 구간을 그대로 "횡보장"으로 승격시키면 강세장 조정 중인
날과 약세장 초입인 날이 뒤섞여 진짜 "추세 없는 박스권"과는 다른 개념이 된다 — 사용자에게 정의 방식을
물었더니 "5분간 인터넷 딥서칭을 통해 스스로 조사해서 정하라"고 위임.

**리서치 결과 채택한 기준**: ADX(Average Directional Index, 추세의 방향이 아니라 세기만 재는 지표)에
대해 TradingView/CrossTrade 등 다수 트레이딩 자료가 공통으로 제시하는 경험칙 — ADX<20은 추세 없음/
횡보(range-bound), 20~25는 추세 형성 애매, 25 이상은 뚜렷한 추세. `core/indicators.py::compute_adx`
신설(`ta.trend.ADXIndicator` 재사용).

**판정 우선순위 재정의** (`core/market_regime.py::classify_daily_regime`, 시그니처가
`close: Series` 하나에서 `close, high, low` 세 개로 바뀜 — ADX 계산에 고가/저가 필요):
1. 52주 고점 대비 낙폭 ≤ -20%면 다른 신호와 무관하게 무조건 "약세장" (기존 규칙 그대로 유지, 최우선).
2. 그 외에 ADX(14) < 20이면 "횡보장" (신규 — 추세 강도 자체가 약함).
3. 나머지(뚜렷한 추세가 있는 날)는 200일선 대비 위치로 "강세장"/"약세장"을 가른다.

기존에 쓰던 `_TREND_CORRECTION_PCT`(200일선 위인데 조정폭 10~20%면 애매하니 버림) 규칙은 완전히
제거됐다 — ADX가 "추세가 있는지 자체"를 직접 재므로 더 이상 임시방편이 필요 없고, 이제 모든 날이
반드시 세 라벨 중 하나로 분류되어(더는 버려지는 잔여 구간이 없음) 학습 데이터 양이 늘어난다.

**연쇄 변경**:
- `core/strategy_tuning.py::_TRAINING_REGIMES`에 "횡보장" 추가 → `run_batch_tuning`이 스타일 그룹마다
  이제 3번(약세장/강세장/횡보장) 따로 트레이닝한다. `historical_regime_segments`도 3개 키를 반환하도록
  확장. 이 두 지점 외에는 기존 인프라(`tune_strategy_for_group`, `save_tuning_run`,
  `trained_regime` DB 컬럼은 원래 자유 문자열이라 스키마 변경 없음)가 이미 국면 개수에 무관하게
  일반화돼 있어 그대로 재사용됨 — UI(1_백테스팅.py 미세튜닝 탭)도 결과에 실제로 존재하는
  (스타일,국면) 조합을 동적으로 순회하므로 카드/표가 자동으로 3개로 늘어난다.
- `select_live_strategy(bear_strategy_id, bull_strategy_id, sideways_strategy_id)` — 세 번째
  인자 추가. 기존에는 `market_regime.select_regime_for_trading()`(4-신호 종합 점수 기반, 대시보드용
  스냅샷 재사용)을 호출했는데, 이 방식은 트레이닝 때 쓰는 `classify_daily_regime`과 완전히 다른
  기준이라 학습-실전 불일치 위험이 있었다 — 그래서 이번에 벤치마크 최근 가격을 직접 가져와
  `classify_daily_regime`을 그대로 재사용하는 방식으로 바꿨다(트레이닝 폴드 구성과 라이브 판단이
  정확히 같은 규칙을 쓰게 됨). `get_market_regime_snapshot`/`select_regime_for_trading`(매크로
  대시보드의 "오늘 국면" 표시용, 4-신호 종합 점수 체계)은 이 절과 무관하게 그대로 둔다 — 서로 다른
  화면의 서로 다른 목적이라 통합하지 않음(과설계 방지).

## 14. 종목 자체 추세 기준 데이터셋 분리 — 상승/하락/횡보 3분류 (2026-07-17, 신규 축)

### 14.1 배경 — 13절의 한계를 사용자가 직접 발견

13절(S&P500 지수 날짜 기준 약세장/강세장)로 실제 튜닝을 돌려본 결과, 조회 구간에 따라 **test
구간(마지막 25%)에 지수 차원의 약세장이 아예 없는 경우**가 실측으로 확인됐다(10년 조회 시 test
구간인 최근 2.5년에 -20% 이상 지수 낙폭이 한 번도 없었음 — 13.6절 정정 논의 중 발견). 이 경우
약세장용 config를 out-of-sample로 검증할 방법이 없다.

사용자가 이 문제를 알게 된 뒤, 지수가 아니라 **종목 자체**의 흐름으로 데이터셋을 나누자는 대안을
직접 제안: "주어진 샘플 기간 동안 하락세였던 티커들로 하락세 알고리즘을, 강세였던 티커로 강세장
알고리즘을, 횡보하는 티커로 횡보장 알고리즘을 연구하고, 실전에서는 임의의 티커가 세 케이스 중
어디인지 판단해 그 전략을 따르게 하라"는 요청. 13절의 "국면은 S&P500 기준, 개별 티커 판단은 안
한다"는 원칙(13.9절 하단)을 사용자가 직접 뒤집은 것 — 새 요청이므로 그대로 반영한다.

**13절과의 관계**: 13절(지수 날짜 기준)을 폐기하지 않고 그대로 둔다 — 두 축은 서로 다른 질문에
답한다("지금 시장 전체가 약세장인가" vs "지금 이 종목이 하락 추세인가"). 이번 14절은 사용자가
직접 요청한 실제 사용 사례(가격+거래량 결합 전략 연구)에 쓰고, 13절은 야간 자동 튜닝 등 기존
용도에 계속 쓸 수 있다.

### 14.2 종목 분류 방법 (기술적 결정, 실측 후 1차 설계에서 수정됨)

**학습 데이터셋 구성 — 절대 임계값이 아니라 상대 순위(3분위)**: 처음엔 `classify_ticker_trend(df)`
(가격 이력 구간 전체 CAGR, ≥+10%="상승"/≤-10%="하락"/나머지="횡보")를 종목별로 독립 적용할
계획이었으나, 실제 10년치 40종목 표본으로 돌려보니 **CAGR<=-10%(하락)인 종목이 0개**였다 —
S&P500 "현재" 구성종목만 표본으로 쓰면 생존편향 때문에(성과가 나빠 지수에서 빠진 종목은 애초에
후보에 없음) 다년 단위 CAGR이 절대적으로 마이너스인 종목이 실질적으로 존재하지 않는다. 그래서
학습 데이터셋 구성용 `classify_tickers_by_trend(tickers, start, end)`는 절대 임계값 대신
**표본 내 CAGR 상대 순위**로 3분위(하위 1/3="하락", 상위 1/3="상승", 중간="횡보")를 나누도록
다시 설계했다 — 시장이 전반적으로 강세였든 아니었든 세 그룹이 항상 채워진다("하락"의 의미가
"절대적으로 손실"이 아니라 "이 표본 안에서 상대적으로 부진했다"로 재정의됨). 유효 종목이 3개
미만이면 3분위 자체가 무의미해 전부 "횡보"로 처리한다.

**라이브 판단은 절대 임계값 그대로 유지**: `classify_ticker_trend(df)`(원래 설계, CAGR
±10% 절대 임계값)는 폐기하지 않고 `select_strategy_for_ticker_trend()`(14.4절, 최근 ~6개월만
보는 라이브 판단)에 그대로 쓴다 — 짧은 구간에서는 개별 종목의 CAGR이 실제로 마이너스가 되는 일이
흔해서(생존편향은 장기·다년 집계에서만 문제가 되는 왜곡) 절대 임계값이 여전히 유효하다. 학습(장기,
상대 순위)과 라이브 판단(단기, 절대 임계값)이 서로 다른 분류 방식을 쓰는 것은 의도된 설계다.

### 14.3 튜닝 방식 — 13절 인프라 재사용, 신규 로직 최소화

13절이 이미 `tune_strategy_for_group(..., regime=None)`(레거시 경로)로 "달력 등분 워크포워드 +
test 구간 전체 평가"를 지원하고 있었다는 점을 이용해, **이 함수 자체는 전혀 손대지 않는다** — 대신
호출하는 쪽에서 종목을 미리 분류해 그 결과로 tickers 인자만 다르게 넘긴다:

1. `classify_tickers_by_trend()`로 표본 전체를 "상승"/"하락"/"횡보" 3그룹으로 나눈다(13절의
   6개 스타일 그룹과는 완전히 별개 축 — 이번 실행에서는 스타일 그룹을 쓰지 않고, `style_type`에
   플레이스홀더 `"전체"`를 넘겨 `_STYLE_PERIOD_MULTIPLIERS`의 기본값(비편향 배수)이 적용되게 함).
2. 그룹별로 `tune_strategy_for_group(group_tickers, base_config, "전체", start, end, regime=None)`
   1회 호출 — `regime=None`이므로 test 구간 전체(마지막 25%)로 평가한다. 날짜를 필터링하는 게
   아니라 **종목을 필터링**했으므로, 그룹에 속한 종목들의 표준 test 구간에는 항상 실제 가격 데이터가
   있다 — 13절이 겪은 "그 국면 데이터가 test에 아예 없음" 문제가 구조적으로 발생하지 않는다.

### 14.4 라이브 선택 — 종목 단위, 최근 구간 기준

`core/strategy_tuning.py::select_strategy_for_ticker_trend(ticker, bear_id, bull_id, sideways_id,
lookback_days=182)` — 13.9절의 `select_live_strategy()`(S&P500 지수 스냅샷 기준)와 별개로 신설.
"지금 시장 전체가 어떤가"가 아니라 "지금 이 종목이 어떤 흐름인가"를 본다. **학습은 표본기간
전체(수년)로, 판단은 최근 약 6개월(`_LIVE_TREND_LOOKBACK_CALENDAR_DAYS=182`, 기존
`_MOMENTUM_LOOKBACK_DAYS`=126거래일과 같은 취지)로** — 의도된 비대칭이다. 학습 목적은 "이 종목이
그 기간 동안 대체로 어떤 흐름을 보였는지"의 깨끗한 신호를 얻는 것이고, 실전 판단 목적은 "지금 당장"
상태를 반영하는 것이라 서로 다른 lookback이 맞다. 종목 자체 가격 이력만 조회하면 되므로(지수 스냅샷
캐시에 의존하지 않음) 매번 즉시 계산 가능 — 별도 사전 계산/캐싱 잡 불필요.

### 14.5 검증

`tests/test_strategy_tuning.py`에 신규 유닛테스트 8개 — CAGR 기준 상승/하락/횡보 분류 경계값,
데이터 부족 시 None, 여러 종목 배치 분류 시 실패 종목 스킵, 라이브 선택이 최근 구간 추세에 맞는
전략을 정확히 고르는지, 가격 조회 실패 시 전부 None. `pytest tests/ -q` 전체 통과.

## 15. 스윙 트레이딩 보유기간 상한 (2026-07-21, 사용자 요청 — 어떤 종목이든 최대 6개월만 보유)

### 15.1 배경

사용자가 스윙 트레이더로서 "어느 주가를 들어도 최대 6개월 정도만 들고 간다"고 확정 — 기존
튜닝 엔진(4~14절)은 보유기간을 전혀 제약하지 않아, 특히 퀄리티 컴파운더/성장주 스타일(넓은
손절, 추세 훼손 전까지 보유)처럼 사실상 1년 이상 들고 가야 나오는 파라미터가 최적으로 뽑힐 수
있었다 — 스윙 트레이더에게는 못 쓰는 결과. 사용자가 추가로 두 가지를 명시적으로 요구: (a) 14절의
상승/하락/횡보 3분류와 반드시 결합할 것, (b) 왜 weight(진입/청산 비중배분)는 항상 고정인지 확인할
것.

### 15.2 보유기간 상한 — `max_holding_days`

`core/backtest_engine.py::_cap_holding_period(position, max_holding_days)` 신규 함수 — 연속
보유(포지션>0) 일수를 하루씩 세다가 상한을 넘기면 그날 강제로 0(청산)으로 덮어쓰고 카운터를
리셋한다. 원래 신호가 그 다음날에도 여전히 보유를 가리키면(예: 롱텀 추세추종 신호가 계속 유지)
바로 재진입한다 — 그 종목을 영구 배제하는 게 아니라 "6개월 넘으면 일단 판다"는 스윙 트레이더의
실제 행동을 모사하는 것.

`run_backtest(..., max_holding_days=)`와 `compare_with_benchmarks(..., max_holding_days=)`에
옵션 인자로 노출했고(기본값 None=제약 없음, 하위호환), `core/strategy_tuning.py`의 그룹 튜닝
호출 체인(`tune_strategy_for_group` → `_select_best_group_config_walkforward` →
`_candidate_group_walkforward_score` → `_candidate_group_train_sharpe`, 그리고 test 평가 쪽
`_evaluate_group_config_on_test` / `_evaluate_group_config_on_regime_matched_test`) 전체에
동일한 `max_holding_days`를 그대로 threading했다 — **탐색 단계(train)부터 이미 이 상한이 강제된
성과로 후보를 고르므로**, 사후 필터링이 아니라 처음부터 스윙 트레이딩 제약 하에서 최적화된다.
`_SWING_MAX_HOLDING_DAYS = 126`(거래일, ≈6개월)을 권장 기본값 상수로 `strategy_tuning.py`에
추가 — 달력일(182일)이 아니라 거래일 기준을 택한 이유는 기존 `_MOMENTUM_LOOKBACK_DAYS`(126거래일,
14절)와 단위를 통일해 두 축이 같은 "6개월"을 가리키게 하기 위함. `compare_with_benchmarks`는
"전략" 쪽에만 상한을 적용하고 매수보유 벤치마크에는 적용하지 않는다(벤치마크는 원래 보유기간
상한이라는 개념이 없는 순수 참고 지표이므로).

### 15.3 국면(횡보/강세/약세) 3분류와의 결합

별도 로직을 새로 만들지 않고, 14절 인프라(`classify_tickers_by_trend`로 나눈 상승/하락/횡보
그룹별로 `tune_strategy_for_group(group_tickers, ..., max_holding_days=126)`을 그대로 호출)에
`max_holding_days`만 항상 넘기면 세 그룹 모두 자동으로 스윙 제약 하에서 탐색된다:

- **횡보**: 원래도 회전율이 높은 전략이 유리해 이 상한의 영향이 작음.
- **상승 추세**: 원래 "추세 끝까지 보유"하는 파라미터가 최적으로 나오기 쉬운데, 상한이 걸리면
  "6개월 안에 최대한 수익을 내고 트레일링으로 잘라내는" 방향으로 탐색이 자연히 이동 — 스윙
  트레이더가 원하는 결과.
- **하락**: 원래도 짧은 숏텀/방어적 파라미터가 유리해 영향이 작음.

즉 13절(지수 날짜 기준 국면)/14절(종목 자체 추세 기준 3분류) 어느 축을 쓰든, 호출부에서
`max_holding_days=_SWING_MAX_HOLDING_DAYS`만 넘기면 국면별 튜닝과 스윙 보유기간 제약이 항상
동시에 적용된다 — 두 축이 서로 배타적이지 않고 항상 함께 작동하도록 설계했다.

### 15.4 weight(진입/청산 비중배분) 탐색 — 이미 구현되어 있었음 (신규 아님)

사용자가 "왜 항상 weight는 고정되는지"를 물었으나 확인 결과 이미 탐색 대상이었다:
`build_param_grid()`(`core/strategy_tuning.py:451`)가 숫자 파라미터 축을 만들 때
`_weight_axis_values()`(같은 파일 382행)를 통해 1:2:6 단계별(staged) 전략의 `entry_stages`/
`exit_stages` 안 weight도 다른 지표 파라미터(이평 기간, RSI 임계값 등)와 동일한 폭으로 독립적으로
흔든다. 다만 weight를 서로 독립적으로 흔들면 합이 1.0(=100%)을 벗어날 수 있어(예: 총 93%만
투자되거나 108% 레버리지), 후보 생성 직후 `_normalize_stage_weights()`(395행)가 같은
entry_stages/exit_stages 안 weight 합을 항상 1.0으로 재정규화한다. 즉 **weight 자체는 이미 매
튜닝마다 변화하고 있고, 사용자가 UI에서 그렇게 안 보였다면 원본 전략이 staged 스키마가 아니거나
(직접 수식/단일 조건 전략은 weight 개념 자체가 없음) 숫자 weight가 1개 이하인 경우(변화시킬
자유도가 없음)일 가능성이 높다** — 코드 결함이 아니라 해당 전략 스키마의 특성.

## 16. 야간 CI 배치의 "잡 통째로 유실" 버그 수정 (2026-07-21)

### 16.1 문제

사용자가 "🌙 리더보드 4.2일 전" 배지를 보고 원인을 물어 조사한 결과, `nightly_tuning.yml`(GitHub
Actions 스케줄)이 2026-07-17~19 사흘 연속 `cancelled`로 끝난 것을 확인:
- 튜닝 루프 예산은 300분인데 잡 자체의 하드 타임리밋(`timeout-minutes`)은 350분뿐이었고, 예산
  체크는 반복 "시작 전"에만 이뤄져 반복 하나(특히 "정밀" 강도, 100종목×5년)가 길어지면 하드
  리밋에 그대로 부딪혀 잡 전체가 강제 종료됐다.
- 더 치명적인 건 `scripts/nightly_tuning_ci.py`가 **모든 반복이 끝난 뒤 딱 한 번만** 리더보드를
  커밋하는 구조였다는 것 — 강제 종료 시점엔 커밋을 한 번도 못 한 상태라 그날 밤 계산한 모든 반복
  결과가 통째로 유실됐다. 결과적으로 리더보드가 07-17 06:35 커밋 이후 07-20 22:08까지 3.65일간
  정체됐다(배지가 보여준 "4.2일 전"의 실제 원인).
- 사용자가 "오라클 VM 같은 상시 서버 없이, GitHub Actions만으로 이 문제를 해결해달라"고 명시적으로
  요청(오라클 인프라는 아직 실제로 쓰지 않고 있어 배제).

### 16.2 해결 — 반복마다 즉시 커밋+푸시

`scripts/nightly_tuning_ci.py`를 "루프가 다 끝나야 커밋" → "**반복 하나가 끝날 때마다 즉시
커밋+푸시**" 구조로 변경:
- `_run_git()`/`_configure_git_identity()`/`_commit_and_push()`/`_save_and_push_leaderboard()`
  신규 헬퍼 — 각 반복 직후 `get_top_tuning_results()`로 최신 DB 상태를 다시 읽어 리더보드 JSON을
  갱신하고, 변경이 있으면(`git diff --cached --quiet`로 판정) 그 자리에서 `git commit` +
  `git push`까지 스크립트 안에서 직접 수행한다.
- 푸시가 거절되면(다른 워크플로/수동 푸시와 경합) `git fetch` + `git rebase` 후 1회 재시도하고,
  그래도 실패하면 `git rebase --abort`로 정리한 뒤 다음 반복에서 다시 시도하도록 하고 계속
  진행한다(커밋 실패가 전체 튜닝 루프를 멈추지 않음).
- 거시지표 캐시 갱신(`_refresh_macro_cache`) 직후에도 한 번 커밋해, 튜닝 루프 진입 직후 죽어도
  캐시 갱신만큼은 유실되지 않게 함.
- 이제 잡이 언제 하드 타임아웃으로 죽든 유실 구간은 "그날 밤 전체"에서 "죽는 순간 진행 중이던
  반복 1개"로 줄어든다.
- `.github/workflows/nightly_tuning.yml`의 기존 "마지막에 한 번 커밋" 스텝은 그대로 두되(위
  변경으로 대부분 no-op가 되는) 폴백으로 남기고, `if: always()`를 추가해 스크립트 스텝이 실패로
  끝나도(잡 자체가 하드킬된 경우는 어차피 이후 스텝이 아예 안 돎 — GitHub Actions의 구조적 한계라
  이건 못 막음) 시도라도 하게 함.
- 예산도 300분 → 270분으로 낮춰(타임리밋 350분 대비 여유 80분) 하드킬 자체가 덜 발생하도록 보수화.
